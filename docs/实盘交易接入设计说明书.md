# A 股低频量化策略研究、回测、模拟盘与实盘执行系统实盘交易接入设计说明书

## 1. 引言

### 1.1 文档目的

本文档用于描述“A 股低频量化策略研究、回测、模拟盘与实盘执行系统”的实盘交易接入设计方案。实盘交易接入模块负责连接真实券商账户，实现账户资金查询、持仓查询、订单提交、撤单、订单状态同步、成交回报处理、异常恢复和审计追踪。

本文档主要回答以下问题：

```text
1. 系统如何接入 A 股券商实盘账户？
2. 券商接口如何抽象？
3. 实盘账户资金和持仓如何同步？
4. 策略信号如何变成真实订单？
5. 实盘订单如何提交、撤单和同步状态？
6. 成交回报如何处理？
7. 订单状态未知时如何恢复？
8. 券商接口断线时如何处理？
9. 如何防止重复下单？
10. 实盘交易如何与风控、监控、审计模块协作？
```

### 1.2 适用范围

本文档适用于以下模块：

```text
broker_gateway        券商接口适配模块
live_account          实盘账户模块
live_position         实盘持仓模块
live_order            实盘订单模块
live_trade            实盘成交模块
live_executor         实盘执行模块
live_risk             实盘风控模块
monitor               监控告警模块
audit                 审计日志模块
```

本文档不适用于：

```text
高频交易接入
极速柜台接入
低延迟专线交易
Tick 级交易
Level-2 盘口交易
做市交易
盘口套利交易
高频撤单交易
```

本系统长期定位为 **A 股低频量化交易系统**，实盘交易接入仅服务于低频策略执行。

## 2. 实盘交易接入总体目标

### 2.1 总体目标

实盘交易接入模块的总体目标是：

> 在严格风控和完整审计的前提下，将经过回测、参数优化、模拟盘验证后的低频策略交易计划，安全地转化为真实券商账户中的委托订单，并持续同步订单、成交、资金和持仓状态。

### 2.2 具体目标

实盘交易接入模块需要实现：

```text
1. 支持连接券商交易接口。
2. 支持查询账户资金。
3. 支持查询账户持仓。
4. 支持查询当日委托。
5. 支持查询当日成交。
6. 支持限价买入。
7. 支持限价卖出。
8. 支持撤单。
9. 支持订单状态同步。
10. 支持成交回报处理。
11. 支持实盘账户资金快照。
12. 支持实盘持仓快照。
13. 支持订单状态异常恢复。
14. 支持券商接口断线重连。
15. 支持重复下单防护。
16. 支持一键停止交易。
17. 支持完整审计日志。
18. 支持实盘小资金灰度运行。
```

## 3. 实盘交易接入设计原则

### 3.1 低频交易原则

本系统不追求毫秒级下单速度，不设计高频交易链路。

实盘接入模块只服务：

```text
日线策略
周线策略
月线策略
低频分钟级辅助执行
收盘后生成交易计划
次日开盘或盘中低频执行
```

不服务：

```text
Tick 级策略
秒级抢单
盘口套利
做市报价
高频撤单
订单簿排队优化
```

### 3.2 策略不直接下单原则

策略模块不得直接调用券商接口。

正确链路：

```text
策略信号
   ↓
交易计划
   ↓
候选订单
   ↓
实盘风控
   ↓
实盘订单模块
   ↓
券商接口适配器
```

错误链路：

```text
策略信号 → 券商接口
```

该链路必须被禁止。

### 3.3 风控前置原则

所有真实订单提交券商前，必须先经过实盘风控检查。

```text
风控不通过 → 不创建真实委托
风控通过 → 才允许提交券商
```

### 3.4 默认关闭原则

实盘交易默认关闭。

系统启动后：

```text
live_trading_enabled = false
```

必须由人工或明确配置开启。

### 3.5 状态同步优先原则

下单前必须同步：

```text
账户资金
账户持仓
可卖数量
当日未完成订单
行情状态
券商连接状态
```

任一关键状态未知时，不允许自动下单。

### 3.6 可追溯原则

每一笔真实订单必须能追溯到：

```text
策略实例
策略参数版本
策略信号
交易计划
风控检查结果
本地订单
券商订单号
成交回报
账户资金变化
持仓变化
审计日志
```

### 3.7 异常保守原则

当系统状态不确定时，默认不交易。

```text
订单状态未知 → 暂停策略
账户同步失败 → 禁止下单
券商接口断开 → 禁止下单
行情异常 → 禁止下单
风控异常 → 禁止下单
```

## 5. 券商接口抽象设计

### 5.1 设计目的

不同券商接口存在差异，系统不能直接依赖某一个券商的具体实现，因此需要抽象统一接口。

系统内部只依赖：

```text
BrokerAdapter
```

不同券商通过实现该接口接入系统。

### 5.2 券商适配器接口

```text
BrokerAdapter
├── connect()
├── disconnect()
├── get_connection_status()
├── query_account()
├── query_positions()
├── query_orders()
├── query_trades()
├── place_order()
├── cancel_order()
└── heartbeat()
```

### 5.3 方法说明

#### connect()

连接券商交易接口。

输入：

```text
broker_account_id
connection_config
```

输出：

```text
connected / failed
```

#### disconnect()

断开券商连接。

#### get_connection_status()

查询当前券商连接状态。

返回：

```text
connected
disconnected
connecting
error
```

#### query_account()

查询账户资金。

返回：

```text
total_asset
available_cash
frozen_cash
market_value
withdrawable_cash
```

#### query_positions()

查询账户持仓。

返回：

```text
symbol
total_quantity
available_quantity
frozen_quantity
today_buy_quantity
cost_price
market_price
market_value
profit_loss
```

#### query_orders()

查询当日委托。

返回：

```text
broker_order_id
symbol
side
price
quantity
filled_quantity
order_status
submit_time
update_time
```

#### query_trades()

查询当日成交。

返回：

```text
broker_trade_id
broker_order_id
symbol
side
price
quantity
amount
commission
tax
trade_time
```

#### place_order()

提交订单。

输入：

```text
symbol
side
price
quantity
order_type
client_order_id
```

输出：

```text
broker_order_id
submit_status
raw_response
```

#### cancel_order()

撤销订单。

输入：

```text
broker_order_id
```

输出：

```text
cancel_status
raw_response
```

#### heartbeat()

用于检查券商接口是否存活。

## 6. 券商接口返回状态标准化

### 6.1 设计目的

券商返回的订单状态可能不同，需要标准化为系统内部订单状态。

### 6.2 系统内部订单状态

```text
created
risk_checked
submitted
accepted
partially_filled
filled
cancel_requested
canceled
rejected
failed
unknown
```

### 6.3 状态映射示例

| 券商原始状态 | 系统状态             |
| ------ | ---------------- |
| 已报     | submitted        |
| 已受理    | accepted         |
| 部成     | partially_filled |
| 已成     | filled           |
| 已撤     | canceled         |
| 废单     | rejected         |
| 拒绝     | rejected         |
| 未知     | unknown          |
| 超时     | unknown          |

### 6.4 状态映射要求

系统应保存：

```text
系统标准状态
券商原始状态
券商原始返回 raw_json
```

原因：

```text
方便排错
方便审计
方便适配不同券商
```

## 7. 实盘账户同步设计

### 7.1 账户同步目标

账户同步用于获取真实账户当前资金状态。

同步内容包括：

```text
总资产
可用资金
冻结资金
持仓市值
可取资金
更新时间
券商原始返回
```

### 7.2 账户同步流程

```text
调用 query_account()
        ↓
获取券商资金数据
        ↓
字段标准化
        ↓
校验数据合法性
        ↓
写入 live_account_snapshot
        ↓
更新账户缓存
        ↓
返回最新账户状态
```

### 7.3 账户同步时机

系统应在以下时机同步账户：

```text
系统启动后
连接券商后
实盘下单前
订单成交后
撤单后
定时任务中
人工手动同步时
异常恢复时
```

### 7.4 下单前强制同步

实盘下单前必须同步账户。

```text
如果账户同步失败 → 拒绝下单
```

### 7.5 账户同步异常

异常包括：

```text
查询失败
返回为空
字段缺失
总资产为负
可用资金为负
持仓市值异常
接口超时
```

处理动作：

```text
1. 标记账户状态异常。
2. 记录风控事件。
3. 发送告警。
4. 禁止自动下单。
```

## 8. 实盘持仓同步设计

### 8.1 持仓同步目标

持仓同步用于获取真实账户当前股票持仓和可卖数量。

同步内容包括：

```text
股票代码
总持仓数量
可卖数量
冻结数量
今日买入数量
成本价
市价
市值
浮动盈亏
```

### 8.2 持仓同步流程

```text
调用 query_positions()
        ↓
获取券商持仓数据
        ↓
字段标准化
        ↓
校验持仓数量
        ↓
写入 live_position_snapshot
        ↓
更新持仓缓存
        ↓
返回最新持仓
```

### 8.3 可卖数量的重要性

A 股有 T+1 规则，因此系统必须使用：

```text
available_quantity
```

而不是：

```text
total_quantity
```

进行卖出校验。

### 8.4 持仓同步时机

```text
系统启动后
连接券商后
实盘下单前
订单成交后
撤单后
定时任务中
订单恢复时
```

### 8.5 持仓同步失败处理

如果持仓同步失败：

```text
1. 拒绝卖出订单。
2. 拒绝新的自动交易。
3. 标记账户状态异常。
4. 记录风控事件。
5. 发送告警。
```

## 9. 实盘交易计划接入设计

### 9.1 交易计划定位

交易计划是策略信号和真实订单之间的中间层。

```text
策略信号 ≠ 真实订单
交易计划 ≠ 真实订单
```

正确流程：

```text
策略信号
   ↓
生成交易计划
   ↓
交易计划预风控
   ↓
人工或系统确认
   ↓
下单前实时风控
   ↓
创建真实订单
```

### 9.2 交易计划字段

```text
plan_id
strategy_instance_id
plan_date
execute_date
plan_status
plan_items
pre_risk_check_result
created_at
updated_at
```

### 9.3 交易计划状态

```text
pending
approved
rejected
executing
executed
canceled
failed
```

### 9.4 交易计划生成流程

```text
读取策略信号
   ↓
读取当前持仓
   ↓
读取目标仓位
   ↓
计算仓位差额
   ↓
生成买卖计划
   ↓
执行预风控
   ↓
保存交易计划
```

### 9.5 交易计划执行前检查

执行前必须检查：

```text
交易计划状态是否 approved
执行日期是否为当前交易日
策略实例是否 running
实盘交易开关是否开启
未触发一键停止
账户和持仓同步成功
```

## 10. 实盘下单设计

### 10.1 实盘下单主流程

```text
读取 approved 交易计划
        ↓
生成候选订单
        ↓
同步账户资金
        ↓
同步账户持仓
        ↓
获取最新行情
        ↓
执行实盘风控
        ↓
风控通过
        ↓
生成 client_order_id
        ↓
创建本地订单 created
        ↓
更新为 risk_checked
        ↓
调用券商 place_order()
        ↓
记录券商返回
        ↓
更新订单状态 submitted / failed / unknown
```

### 10.2 下单前必须满足

```text
1. 实盘交易开关已开启。
2. 未触发一键停止。
3. 策略实例处于 running。
4. 交易计划已 approved。
5. 当前日期为交易日。
6. 当前时间在允许交易时段。
7. 券商接口 connected。
8. 行情状态正常。
9. 账户同步成功。
10. 持仓同步成功。
11. 风控检查通过。
12. 不存在重复下单风险。
```

### 10.3 本地订单号设计

系统应生成唯一的 client_order_id。

建议格式：

```text
client_order_id =
strategy_instance_id + plan_id + symbol + side + execute_date + hash
```

示例：

```text
100-9001-600519.SH-buy-20260522-a8f3
```

数据库中必须对 client_order_id 建唯一索引。

### 10.4 订单创建顺序

推荐顺序：

```text
1. 先通过风控。
2. 创建本地订单。
3. 记录 risk_check_result。
4. 调用券商接口。
5. 记录券商返回。
6. 更新订单状态。
```

### 10.5 下单接口输入

```text
broker_account_id
strategy_instance_id
plan_id
signal_id
symbol
side
price
quantity
order_type
```

### 10.6 下单接口输出

成功：

```json
{
  "live_order_id": 7001,
  "client_order_id": "100-9001-600519.SH-buy-20260522-a8f3",
  "broker_order_id": "B202605220001",
  "order_status": "submitted"
}
```

失败：

```json
{
  "order_status": "failed",
  "error_message": "broker order rejected"
}
```

未知：

```json
{
  "order_status": "unknown",
  "error_message": "broker order timeout, need recovery"
}
```

## 11. 实盘撤单设计

### 11.1 撤单适用场景

撤单适用于：

```text
订单长时间未成交
交易计划取消
人工要求撤单
风控要求停止交易
价格明显偏离
策略暂停
```

### 11.2 撤单流程

```text
查询本地订单
        ↓
检查订单是否可撤
        ↓
调用券商 cancel_order()
        ↓
记录撤单请求
        ↓
更新订单状态 cancel_requested
        ↓
同步券商订单状态
        ↓
更新为 canceled / failed / unknown
```

### 11.3 可撤订单状态

以下状态允许撤单：

```text
submitted
accepted
partially_filled
```

以下状态不允许撤单：

```text
filled
canceled
rejected
failed
unknown
```

unknown 状态需要先恢复状态，再决定是否撤单。

### 11.4 撤单失败处理

撤单失败时：

```text
1. 记录错误。
2. 同步券商订单状态。
3. 如果无法确认，标记 unknown。
4. 发送告警。
5. 等待人工处理。
```

## 12. 实盘订单状态同步设计

### 12.1 同步目标

订单状态同步用于保证本地订单状态与券商订单状态一致。

同步内容：

```text
券商订单号
订单状态
委托数量
成交数量
剩余数量
拒单原因
更新时间
```

### 12.2 同步方式

支持两种方式：

```text
主动查询
券商推送回报
```

第一版建议先实现主动查询：

```text
定时 query_orders()
定时 query_trades()
```

### 12.3 同步时机

```text
下单后立即同步
撤单后立即同步
交易时段定时同步
系统启动后同步
异常恢复时同步
收盘后同步
```

### 12.4 同步流程

```text
调用 query_orders()
        ↓
获取券商当日委托
        ↓
按 broker_order_id 或 client_order_id 匹配本地订单
        ↓
标准化订单状态
        ↓
更新本地订单
        ↓
记录状态变化日志
        ↓
发现 unknown 则触发恢复流程
```

### 12.5 状态变化审计

订单状态变化必须记录：

```text
order_id
old_status
new_status
broker_order_id
raw_status
updated_at
```

## 13. 成交回报处理设计

### 13.1 成交回报目标

成交回报用于记录真实成交，并更新订单、资金和持仓。

### 13.2 成交流程

```text
调用 query_trades()
        ↓
获取券商成交记录
        ↓
成交记录标准化
        ↓
成交去重
        ↓
匹配本地订单
        ↓
写入 live_trade
        ↓
更新订单成交数量
        ↓
更新订单状态
        ↓
触发账户和持仓同步
        ↓
记录审计日志
```

### 13.3 成交去重

优先使用：

```text
broker_account_id + broker_trade_id
```

如果券商没有稳定成交编号，可使用组合字段：

```text
broker_order_id + symbol + side + price + quantity + trade_time
```

### 13.4 成交对订单状态的影响

```text
filled_quantity = sum(trade.quantity)
remaining_quantity = order.quantity - filled_quantity
```

如果：

```text
filled_quantity == 0
```

订单保持 submitted 或 accepted。

如果：

```text
0 < filled_quantity < order.quantity
```

订单状态为：

```text
partially_filled
```

如果：

```text
filled_quantity == order.quantity
```

订单状态为：

```text
filled
```

### 13.5 成交后账户同步

成交后必须重新同步：

```text
账户资金
账户持仓
可卖数量
```

原因：

```text
本地计算可能和券商实际扣费、冻结、成交细节存在差异。
```

## 14. 订单状态未知恢复设计

### 14.1 unknown 状态来源

订单状态 unknown 可能来源于：

```text
下单请求超时
券商返回超时
网络中断
程序崩溃
券商连接断开
提交成功但本地未收到回执
撤单请求结果不确定
```

### 14.2 unknown 状态原则

一旦出现 unknown：

```text
禁止继续自动下单
暂停相关策略
记录 critical 风控事件
发送告警
```

### 14.3 恢复流程

```text
发现 unknown 订单
        ↓
暂停相关策略
        ↓
查询券商当日委托
        ↓
查询券商当日成交
        ↓
尝试通过 client_order_id 匹配
        ↓
尝试通过 broker_order_id 匹配
        ↓
尝试通过股票、方向、价格、数量、时间匹配
        ↓
匹配成功：修正订单状态
        ↓
匹配失败：保持 unknown
        ↓
等待人工处理
```

### 14.4 恢复后的处理

如果恢复为 submitted 或 accepted：

```text
继续同步订单状态，但不自动补单。
```

如果恢复为 filled：

```text
写入成交记录，更新账户和持仓。
```

如果恢复为 rejected 或 failed：

```text
记录原因，不自动重试。
```

如果仍为 unknown：

```text
保持策略暂停，等待人工确认。
```

## 15. 重复下单防护设计

### 15.1 重复下单风险来源

```text
接口重试
任务重复调度
网络超时
程序重启
交易计划重复执行
策略重复生成信号
订单状态 unknown 后错误补单
```

### 15.2 防护机制

系统必须具备：

```text
client_order_id 唯一
plan_id 执行唯一
signal_id 关联
同一计划项只能执行一次
同股票同方向未完成订单检查
短时间重复下单检查
下单前同步当日委托
程序启动后先恢复订单状态
```

### 15.3 数据库约束

live_order 表中：

```text
client_order_id UNIQUE
```

交易计划项也应有唯一执行标记：

```text
plan_id + symbol + side UNIQUE 或应用层唯一校验
```

### 15.4 超时重试原则

下单请求超时时，不得立即再次提交相同订单。

正确处理：

```text
1. 标记订单 unknown。
2. 查询券商委托。
3. 查询券商成交。
4. 确认是否已提交。
5. 确认失败后等待人工处理。
```

## 16. 券商连接管理设计

### 16.1 连接状态

券商连接状态包括：

```text

error
reconnecting
```

### 16.2 连接流程

```text
读取券商账户配置
        ↓
初始化适配器
        ↓
调用 connect()
        ↓
执行 heartbeat()
        ↓
连接成功
        ↓
同步账户和持仓
```

### 16.3 心跳机制

系统应定时调用：

```text
heartbeat()
```

或轻量查询接口，确认连接状态。

### 16.4 断线处理

当券商接口断开：

```text
1. 标记 broker_status = disconnected。
2. 停止自动下单。
3. 保留已有订单状态。
4. 尝试重连。
5. 记录告警。
6. 重连后执行订单恢复。
```

### 16.5 重连流程

```text
检测断线
   ↓
停止新下单
   ↓
尝试 reconnect
   ↓
连接成功
   ↓
同步账户
   ↓
同步持仓
   ↓
同步当日委托
   ↓
同步当日成交
   ↓
检查 unknown 订单
   ↓
人工或系统恢复交易
```

### 16.6 重连后默认不自动恢复交易

建议第一版重连后不自动恢复交易。

需要满足：

```text
账户同步成功
持仓同步成功
订单状态确认
无 unknown 订单
人工确认
```

才可恢复自动交易。

## 19. A 股交易规则接入设计

### 19.1 交易日检查

实盘下单前检查：

```text
is_trading_day(today) == true
```

否则拒绝。

### 19.2 交易时段检查

默认允许：

```text
09:30 - 11:30
13:00 - 15:00
```

是否允许集合竞价可配置。

### 19.3 停牌检查

```text
is_suspended(symbol) == false
```

停牌则拒绝买入和卖出。

### 19.4 涨跌停检查

买入价格：

```text
price <= limit_up_price
```

卖出价格：

```text
price >= limit_down_price
```

超出范围拒绝。

### 19.5 T+1 检查

卖出数量必须满足：

```text
quantity <= available_quantity
```

不得使用 total_quantity 代替 available_quantity。

### 19.6 买入单位检查

买入数量应符合：

```text
quantity % 100 == 0
```

不符合则拒绝或由仓位模块提前调整。

第一版建议拒绝不合规订单。

```text
        else:

    sync orders
    sync trades
    update plan status
```

```text
券商接口稳定性

订单状态同步准确性

审计日志完整性
异常恢复能力
```

### 28.2 灰度限制

建议初期限制：

```text
总资金上限较低
单笔订单金额较低
单票仓位较低
每日订单数较低
每日成交金额较低
只运行一个策略

必须人工监控
```

## 29. 实盘异常场景设计

### 29.1 券商接口断开

处理：

```text
1. 停止新下单。
2. 标记券商状态 disconnected。
3. 发送告警。
4. 尝试重连。
5. 重连后同步订单和成交。
6. 存在 unknown 订单时等待人工确认。
```

### 29.2 下单请求超时

处理：

```text
1. 本地订单标记 unknown。
2. 暂停相关策略。
3. 查询券商委托。
4. 查询券商成交。
5. 不允许立即重试下单。
```

### 29.3 账户同步失败

处理：

```text
1. 拒绝新订单。
2. 标记账户异常。
3. 发送告警。
4. 尝试重新同步。
```

### 29.4 持仓同步失败

处理：

```text
1. 拒绝卖出订单。
2. 拒绝新自动交易。
3. 发送告警。
4. 等待同步恢复。
```

### 29.5 风控模块异常

处理：

```text
1. 拒绝所有实盘订单。
2. 标记风控异常。
3. 发送 critical 告警。
4. 等待人工处理。
```

### 29.6 重复下单风险

处理：

```text
1. 拒绝当前订单。
2. 记录风控事件。
3. 暂停相关交易计划。
4. 必要时暂停策略。
```

### 29.7 部分成交

处理：

```text
1. 更新订单为 partially_filled。
2. 记录已成交数量。
3. 同步账户和持仓。
4. 剩余部分继续等待或根据规则撤单。
```

### 29.8 撤单失败

处理：

```text
1. 同步券商订单状态。
2. 如果订单已成交，更新为 filled。
3. 如果状态未知，标记 unknown。
4. 发送告警。
```

## 32. 验收标准

### 32.1 券商接口验收

系统应满足：

```text
1. 能连接券商接口。
2. 能断开券商接口。
3. 能查询连接状态。
4. 能查询账户资金。
5. 能查询账户持仓。
6. 能查询当日委托。
7. 能查询当日成交。
```

### 32.2 下单撤单验收

系统应满足：

```text
1. 风控通过后能提交买入委托。
2. 风控通过后能提交卖出委托。
3. 风控拒绝时不调用券商接口。
4. 能撤销可撤订单。
5. 撤单后能同步订单状态。
```

### 32.3 订单成交验收

系统应满足：

```text
1. 能记录本地订单。
2. 能记录券商订单号。
3. 能同步订单状态。
4. 能处理部分成交。
5. 能处理全部成交。
6. 能处理拒单。
7. 能处理撤单。
8. 能处理 unknown 订单。
9. 能记录成交回报。
10. 成交不会重复入库。
```

### 32.4 账户持仓验收

系统应满足：

```text
1. 下单前强制同步账户。
2. 下单前强制同步持仓。
3. 成交后重新同步账户。
4. 成交后重新同步持仓。
5. 可卖数量使用券商返回值。
6. 账户同步失败时禁止下单。
```

### 32.5 风控与安全验收

系统应满足：

```text
1. 实盘交易默认关闭。
2. 一键停止后拒绝所有新订单。
3. 券商断线时拒绝下单。
4. 行情异常时拒绝下单。
5. 风控异常时拒绝下单。
6. 重复订单被拒绝。
7. 策略异常频率被暂停。
8. 所有实盘关键操作有审计日志。
```

## 34. 推荐实盘执行流程

### 34.1 收盘后

```text
1. 更新当日行情。
2. 计算指标和因子。
3. 策略生成信号。
4. 生成次日交易计划。
5. 执行预风控。
6. 保存交易计划。
```

### 34.2 次日开盘前

```text
1. 连接券商接口。
2. 同步账户资金。
3. 同步账户持仓。
4. 同步当日委托。
5. 检查实盘交易开关。
6. 检查风控状态。
7. 检查交易计划状态。
```

### 34.3 交易时段

```text
1. 获取最新行情。
2. 执行下单前风控。
3. 提交订单。
4. 同步订单状态。
5. 同步成交回报。
6. 更新资金和持仓。
7. 记录审计日志。
```

### 34.4 收盘后

```text
1. 同步最终订单状态。
2. 同步最终成交。
3. 同步账户资金。
4. 同步持仓。
5. 生成实盘日报。
6. 记录异常和风控事件。
7. 复盘策略执行情况。
```

## 35. 总结

本文档完成了 A 股低频量化策略研究、回测、模拟盘与实盘执行系统的实盘交易接入设计。

实盘交易接入模块的核心职责是：

```text
连接券商接口
查询账户资金
查询账户持仓
生成实盘订单
提交买入卖出委托
撤单
同步订单状态
处理成交回报
处理异常和恢复
记录审计日志
```

系统实盘接入的核心原则是：

```text
策略不能直接下单；
交易计划不等于订单；
实盘订单必须经过风控；
状态不确定时禁止交易；
订单 unknown 时暂停策略；
下单前必须同步账户和持仓；
成交后必须同步账户和持仓；
所有关键操作必须审计；
系统长期只支持低频实盘交易。
```

最终，实盘交易接入模块应保证：

```text
能安全连接券商；
能准确同步账户；
能准确同步持仓；
能受控提交订单；
能处理成交和撤单；
能避免重复下单；
能在异常时停止交易；
能让每一笔实盘交易都可追溯。
```

回测通过不代表可以直接实盘。策略必须依次经过：

```text
历史回测
样本外验证
参数稳定性检查
模拟盘运行
Mock 券商测试
小资金灰度实盘
正式实盘
```

实盘交易接入模块是系统从“研究工具”进入“真实资金执行系统”的关键边界，因此必须以稳定性、安全性、风控和可追溯性为最高优先级。
