# A 股低频量化策略研究、回测、模拟盘与实盘执行系统数据库设计说明书

## 1. 引言

### 1.1 文档目的

本文档用于描述“A 股低频量化策略研究、回测、模拟盘与实盘执行系统”的数据库设计方案。本文档在软件需求规格说明书（SRS）和系统概要设计说明书的基础上，进一步细化系统的数据模型、表结构、字段定义、主外键关系、索引设计、约束设计、数据一致性设计、数据生命周期和数据库使用规范。

本文档主要回答以下问题：

1. 系统需要哪些核心数据表？
2. 各数据表分别存储什么内容？
3. 表与表之间如何关联？
4. 行情、指标、因子、策略、回测、模拟盘、实盘交易如何落库？
5. 实盘订单、成交、资金、持仓如何保证可追溯？
6. 如何设计索引以支持常用查询？
7. 如何通过约束避免重复数据、重复订单和状态混乱？
8. 如何保证系统数据库设计符合低频量化交易定位？

### 1.2 适用范围

本文档适用于以下模块：

* 股票基础数据模块
* 历史行情数据模块
* 可选分钟级行情模块
* 交易日历模块
* 技术指标模块
* 因子模块
* 策略管理模块
* 策略参数模块
* 回测模块
* 参数优化模块
* 模拟盘模块
* 券商账户模块
* 实盘账户快照模块
* 实盘持仓快照模块
* 实盘订单模块
* 实盘成交模块
* 风控事件模块
* 审计日志模块
* 告警模块

本文档不包含 Tick 级行情表、Level-2 盘口表、逐笔成交表、高频订单簿表或高频撮合相关表。系统长期不支持高频交易，因此数据库设计中不建设高频数据链路。

### 1.3 数据库选型

系统推荐使用 PostgreSQL 作为主数据库。

选择 PostgreSQL 的原因：

1. 支持事务，适合订单、成交、资金、持仓等一致性要求较高的数据。
2. 支持 JSONB，适合保存策略参数、风控配置、接口返回详情等半结构化数据。
3. 支持丰富索引类型，适合行情和交易数据查询。
4. 支持复杂 SQL 查询，便于统计报表和回测结果分析。
5. 生态成熟，适合后端系统长期维护。

Redis 可作为缓存、分布式锁和任务状态辅助存储，但不能作为实盘订单、成交、资金、持仓等关键数据的唯一存储。

### 1.4 设计原则

数据库设计遵循以下原则：

1. **低频定位原则**：系统不设计 Tick 数据表，不设计 Level-2 盘口表，不建设高频交易数据链路。
2. **可追溯原则**：策略信号、交易计划、风控检查、订单、成交、资金、持仓、人工操作均应可追溯。
3. **一致性原则**：实盘订单和成交等关键数据应通过事务保证状态一致。
4. **唯一性原则**：行情数据、订单编号、成交编号等关键字段应通过唯一索引防止重复。
5. **扩展性原则**：策略参数、风控配置、券商返回详情等允许使用 JSONB 存储，方便后续扩展。
6. **查询效率原则**：高频查询字段必须建立索引，如 symbol、trade_date、task_id、strategy_id、order_status 等。
7. **审计优先原则**：实盘相关操作必须记录审计日志，不能只更新最终状态。
8. **读写分离预留原则**：第一版可使用单库，后续可扩展只读副本用于报告查询。

## 3. 数据库命名规范

### 3.1 表命名规范

1. 表名使用小写字母和下划线。
2. 表名使用业务含义清晰的英文名。
3. 表名尽量使用单数或领域名词组合。
4. 实盘相关表统一使用 `live_` 前缀。
5. 模拟盘相关表统一使用 `simulation_` 前缀。
6. 回测相关表统一使用 `backtest_` 前缀。
7. 优化相关表统一使用 `optimize_` 前缀。

示例：

```text
stock_info
stock_daily
strategy
backtest_task
simulation_order
live_order
live_trade
live_audit_log
```

### 3.2 字段命名规范

1. 字段名使用小写字母和下划线。
2. 主键统一命名为 `id`。
3. 外键使用 `{关联表名}_id`。
4. 时间字段统一使用 `_at` 或 `_time` 后缀。
5. JSON 字段使用 `_json` 后缀。
6. 布尔字段使用 `is_` 或 `has_` 前缀。

示例：

```text
strategy_id
created_at
updated_at
params_json
is_suspended
has_error
```

### 3.3 通用字段规范

大多数业务表建议包含：

```text
id
created_at
updated_at
```

部分只追加写入的日志表可以只包含：

```text
id
created_at
```

### 3.4 数据类型规范

| 类型        | 使用场景              |
| --------- | ----------------- |
| bigint    | 主键、外键、大数量 ID      |
| varchar   | 股票代码、状态、类型、名称等短文本 |
| text      | 长描述、错误信息、备注       |
| numeric   | 金额、价格、收益率、比例等金融数值 |
| integer   | 数量、次数、轮次等整数       |
| boolean   | 是否停牌、是否涨停、是否启用等   |
| date      | 交易日期、上市日期等        |
| timestamp | 创建时间、更新时间、成交时间等   |
| jsonb     | 策略参数、风控配置、接口返回详情等 |

金融金额和价格不建议使用 float，应使用 numeric，避免浮点误差。

## 5. 基础数据域表设计

## 5.1 股票基础信息表：stock_info

### 5.1.1 表说明

`stock_info` 用于存储 A 股股票基础信息，包括股票代码、名称、交易所、行业、上市日期和状态。

### 5.1.2 字段设计

| 字段名         | 类型           | 约束               | 说明                          |
| ----------- | ------------ | ---------------- | --------------------------- |
| id          | bigint       | PK               | 主键                          |
| symbol      | varchar(32)  | UNIQUE, NOT NULL | 股票代码，如 600519.SH            |
| name        | varchar(128) | NOT NULL         | 股票名称                        |
| exchange    | varchar(16)  | NOT NULL         | 交易所，如 SH、SZ、BJ              |
| market      | varchar(32)  | NULL             | 市场板块，如 main、star、gem        |
| industry    | varchar(128) | NULL             | 所属行业                        |
| list_date   | date         | NULL             | 上市日期                        |
| delist_date | date         | NULL             | 退市日期                        |
| status      | varchar(32)  | NOT NULL         | listed、delisted、suspended 等 |
| created_at  | timestamp    | NOT NULL         | 创建时间                        |
| updated_at  | timestamp    | NOT NULL         | 更新时间                        |

### 5.1.3 索引设计

| 索引名                     | 字段       | 类型     | 说明       |
| ----------------------- | -------- | ------ | -------- |
| uk_stock_info_symbol    | symbol   | UNIQUE | 防止股票代码重复 |
| idx_stock_info_exchange | exchange | NORMAL | 按交易所查询   |
| idx_stock_info_industry | industry | NORMAL | 按行业查询    |
| idx_stock_info_status   | status   | NORMAL | 按状态查询    |

### 5.1.4 示例 SQL

```sql
CREATE TABLE stock_info (
    id BIGSERIAL PRIMARY KEY,
    symbol VARCHAR(32) NOT NULL UNIQUE,
    name VARCHAR(128) NOT NULL,
    exchange VARCHAR(16) NOT NULL,
    market VARCHAR(32),
    industry VARCHAR(128),
    list_date DATE,
    delist_date DATE,
    status VARCHAR(32) NOT NULL DEFAULT 'listed',
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## 6. 行情数据域表设计

## 6.1 日线行情表：stock_daily

### 6.1.1 表说明

`stock_daily` 用于存储股票日线行情数据，是系统最核心的数据表之一。

### 6.1.2 字段设计

| 字段名              | 类型            | 约束       | 说明   |
| ---------------- | ------------- | -------- | ---- |
| id               | bigint        | PK       | 主键   |
| symbol           | varchar(32)   | NOT NULL | 股票代码 |
| trade_date       | date          | NOT NULL | 交易日期 |
| open             | numeric(18,4) | NULL     | 开盘价  |
| high             | numeric(18,4) | NULL     | 最高价  |
| low              | numeric(18,4) | NULL     | 最低价  |
| close            | numeric(18,4) | NULL     | 收盘价  |
| pre_close        | numeric(18,4) | NULL     | 前收盘价 |
| volume           | numeric(24,4) | NULL     | 成交量  |
| amount           | numeric(24,4) | NULL     | 成交额  |
| adj_factor       | numeric(18,8) | NULL     | 复权因子 |
| turnover_rate    | numeric(18,8) | NULL     | 换手率  |
| limit_up_price   | numeric(18,4) | NULL     | 涨停价  |
| limit_down_price | numeric(18,4) | NULL     | 跌停价  |
| is_suspended     | boolean       | NOT NULL | 是否停牌 |
| is_limit_up      | boolean       | NOT NULL | 是否涨停 |
| is_limit_down    | boolean       | NOT NULL | 是否跌停 |
| source           | varchar(64)   | NULL     | 数据来源 |
| created_at       | timestamp     | NOT NULL | 创建时间 |
| updated_at       | timestamp     | NOT NULL | 更新时间 |

### 6.1.3 约束设计

1. `(symbol, trade_date)` 必须唯一。
2. high 不应小于 low。
3. open、high、low、close 不应为负数。
4. volume 和 amount 不应为负数。

### 6.1.4 索引设计

| 索引名                        | 字段                 | 类型     | 说明         |
| -------------------------- | ------------------ | ------ | ---------- |
| uk_stock_daily_symbol_date | symbol, trade_date | UNIQUE | 防止重复行情     |
| idx_stock_daily_date       | trade_date         | NORMAL | 按日期查询全市场行情 |
| idx_stock_daily_symbol     | symbol             | NORMAL | 按股票查询历史行情  |
| idx_stock_daily_suspended  | is_suspended       | NORMAL | 查询停牌数据     |

### 6.1.5 示例 SQL

```sql
CREATE TABLE stock_daily (
    id BIGSERIAL PRIMARY KEY,
    symbol VARCHAR(32) NOT NULL,
    trade_date DATE NOT NULL,
    open NUMERIC(18,4),
    high NUMERIC(18,4),
    low NUMERIC(18,4),
    close NUMERIC(18,4),
    pre_close NUMERIC(18,4),
    volume NUMERIC(24,4),
    amount NUMERIC(24,4),
    adj_factor NUMERIC(18,8),
    turnover_rate NUMERIC(18,8),
    limit_up_price NUMERIC(18,4),
    limit_down_price NUMERIC(18,4),
    is_suspended BOOLEAN NOT NULL DEFAULT FALSE,
    is_limit_up BOOLEAN NOT NULL DEFAULT FALSE,
    is_limit_down BOOLEAN NOT NULL DEFAULT FALSE,
    source VARCHAR(64),
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE (symbol, trade_date),
    CHECK (high IS NULL OR low IS NULL OR high >= low),
    CHECK (volume IS NULL OR volume >= 0),
    CHECK (amount IS NULL OR amount >= 0)
);
```

## 8. 策略域表设计

## 8.1 策略定义表：strategy

### 8.1.1 表说明

`strategy` 用于存储策略基础定义。策略定义与策略参数分离。

### 8.1.2 字段设计

| 字段名         | 类型           | 约束       | 说明                                        |

| id          | bigint       | PK       | 主键                                        |

| description | text         | NULL     | 策略描述                                      |
| version     | varchar(64)  | NOT NULL | 策略代码版本                                    |
| status      | varchar(32)  | NOT NULL | active、disabled                           |
| created_at  | timestamp    | NOT NULL | 创建时间                                      |
| updated_at  | timestamp    | NOT NULL | 更新时间                                      |

### 8.1.3 约束设计

## 15. 数据一致性设计

### 15.1 关键一致性要求

系统必须保证以下一致性：

1. 策略参数版本与回测结果一致。
2. 回测任务与回测结果一一对应。
3. 实盘订单与成交记录可关联。
4. 成交记录不得重复处理。
5. 实盘订单状态不得非法跳转。
6. 资金快照和持仓快照应反映券商账户状态。
7. 风控拒绝的订单不得提交券商。
8. 实盘启停和参数变更必须审计。

### 15.2 事务设计

以下操作必须使用数据库事务：

1. 创建实盘订单和记录风控检查结果。
2. 写入成交记录和更新订单状态。
3. 写入交易计划和预风控结果。
4. 更新实盘开关和写入审计日志。
5. 一键停止交易和写入风控事件。
6. 恢复订单状态和写入审计日志。

### 15.3 防重复设计

#### 15.3.1 行情重复

通过 `(symbol, trade_date)` 唯一索引防止日线行情重复。

#### 15.3.2 参数重复

通过 `(strategy_id, param_hash)` 唯一索引防止相同参数重复保存。

#### 15.3.3 订单重复

通过 `client_order_id` 唯一索引防止本地重复下单。

#### 15.3.4 成交重复

如果券商提供 broker_trade_id，通过 `(broker_account_id, broker_trade_id)` 防止成交重复处理。

### 15.4 状态机约束

订单状态应按合法路径流转：

```text
created → risk_checked → submitted → accepted → partially_filled → filled
created → rejected
submitted → failed
accepted → cancel_requested → canceled
任意关键异常 → unknown
```

状态机校验可在应用层实现，数据库层保存最终状态和变更日志。

## 17. 数据生命周期设计

### 17.1 行情数据

日线行情、指标和因子属于长期数据，应长期保存。

### 17.2 回测数据

回测任务和回测结果应长期保存。大量回测交易明细可根据存储成本设定归档策略。

### 17.3 优化数据

优化任务和最优结果应长期保存。trial 明细可根据任务规模归档。

### 17.4 模拟盘数据

模拟盘订单、成交和资产曲线应至少保存到策略验证周期结束。重要模拟结果应长期保存。

### 17.5 实盘数据

实盘订单、成交、资金快照、持仓快照、风控事件和审计日志必须长期保存，不建议删除。可按年度归档。

### 17.6 告警数据

告警数据可按时间归档，已解决告警可保留摘要。

## 18. 低频交易边界设计

### 18.1 不建设的表

为保证项目长期不向高频交易扩展，数据库不设计以下表：

```text
tick_trade
level2_order_book
level2_snapshot
order_book_queue
hft_order_event
hft_latency_log
market_making_quote
```

### 18.2 可选分钟行情的限制

`stock_minute_optional` 仅用于：

1. 盘中价格确认。
2. 模拟盘辅助成交。
3. 实盘风控检查。
4. 低频策略辅助观察。

不得用于：

1. 高频策略生成。
2. 盘口套利。
3. 做市报价。
4. 毫秒级下单。

### 18.3 策略频率字段约束

`strategy.frequency` 只允许：

```text
daily
weekly
monthly
low_frequency_minute
```

不允许：

```text
tick
hft
level2
millisecond
microsecond
```

建议：

1. 每日全量备份数据库。
2. 交易日盘中可增加增量备份或 WAL 归档。
3. 实盘订单和成交数据应重点保护。
4. 备份文件应定期恢复演练。

### 19.3 恢复要求

系统恢复后应：

1. 恢复策略实例状态。
2. 恢复未完成订单。
3. 重新同步券商账户。
4. 重新同步当日委托和成交。
5. 对不确定状态订单标记为 unknown。
6. 暂停自动交易，等待确认。

## 20. 数据库安全设计

### 20.1 敏感信息保护

数据库中不得明文保存：

1. 券商交易密码。
2. API 密钥。
3. 登录验证码。
4. 完整账户号。

### 20.2 脱敏设计

券商账户号应保存脱敏版本，例如：

```text
1234******5678
```

### 20.3 权限设计预留

后续应支持不同数据库角色：

| 角色             | 权限      |
| -------------- | ------- |
| app_rw         | 应用读写权限  |
| app_readonly   | 报表只读权限  |
| migration_user | 数据库迁移权限 |
| admin          | 管理权限    |

### 20.4 审计保护

审计日志不应被普通业务流程随意修改或删除。后续可将审计日志写入单独库或追加式日志系统。

## 23. 总结

本文档完成了 A 股低频量化策略研究、回测、模拟盘与实盘执行系统的数据库设计。

系统数据库设计围绕以下核心目标展开：

1. 支持 A 股低频策略研究。
2. 支持历史行情、指标、因子和策略参数存储。
3. 支持回测任务、回测结果和交易记录存储。
4. 支持参数优化过程追踪。
5. 支持模拟盘账户、订单、成交和持仓管理。
6. 支持实盘券商账户、资金快照、持仓快照、订单和成交记录。
7. 支持实盘风控事件、告警和审计日志。
8. 支持订单、成交、资金和持仓的可追溯性。
9. 明确不建设高频交易相关数据表和数据链路。

该数据库设计可作为后续数据库建模、SQL 编写、ORM 映射、接口开发、测试数据构造和系统实现的基础。
