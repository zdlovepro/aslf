# A 股低频量化策略研究、回测、模拟盘与实盘执行系统概要设计说明书

## 1. 引言

### 1.1 文档目的

本文档用于描述“A 股低频量化策略研究、回测、模拟盘与实盘执行系统”的总体技术设计方案。本文档在软件需求规格说明书（SRS）的基础上，进一步说明系统如何实现，包括系统架构、模块划分、核心流程、数据流转、接口协作、数据库设计思路、风控介入点、异常处理机制、部署方式和后续扩展方向。

本文档主要回答以下问题：

1. 系统整体架构如何设计？
2. 系统由哪些核心模块组成？
3. 各模块之间如何交互？
4. 策略信号如何从研究阶段进入回测、模拟盘和实盘？
5. 实盘订单如何经过风控并提交到券商接口？
6. 系统如何保证资金、持仓、订单和成交状态一致？
7. 系统如何处理异常、断线、重复下单和订单状态未知等问题？
8. 系统如何保证长期只面向低频交易，不扩展为高频交易系统？

本文档可作为后续详细设计、编码实现、接口设计、数据库设计、测试设计和部署实施的依据。

### 1.2 适用范围

本文档适用于以下系统范围：

* A 股历史数据管理
* 日线及可选分钟级低频行情处理
* 技术指标计算
* 量化因子计算
* 低频策略开发与管理
* 历史回测
* 参数优化
* 模拟盘运行
* 券商交易接口接入
* 实盘账户同步
* 实盘订单管理
* 实盘成交回报处理
* 实盘风控
* A 股交易规则适配
* 监控告警
* 审计日志
* 后端 API 服务

本文档不适用于以下系统：

* 高频交易系统
* Tick 级策略系统
* Level-2 盘口策略系统
* 做市交易系统
* 高频撮合仿真系统
* 毫秒级或微秒级低延迟交易系统

### 1.3 设计目标

系统概要设计目标如下：

1. 构建一个模块清晰、职责明确、可扩展的低频量化交易后端架构。
2. 支持从策略研究到实盘执行的完整闭环。
3. 保证策略、回测、模拟盘和实盘之间逻辑尽可能一致。
4. 保证所有实盘订单必须经过风控检查。
5. 保证资金、持仓、订单、成交和审计记录可追溯。
6. 保证系统异常时能够及时停止自动交易。
7. 保证系统不会向高频交易方向扩展。
8. 为后续数据库设计、API 设计、风控设计、测试设计提供依据。

### 1.4 设计原则

系统设计遵循以下原则：

#### 1.4.1 低频优先原则

系统长期定位为 A 股低频量化系统，核心策略频率以日线、周线、月线和低频分钟级策略为主。系统不设计 Tick 级数据链路，不设计高频撮合模型，不追求毫秒级低延迟下单。

#### 1.4.2 策略与交易解耦原则

策略模块只负责生成信号或目标持仓，不允许直接调用券商接口。所有实盘订单必须经过交易计划模块、风控模块和订单管理模块。

#### 1.4.3 回测、模拟盘、实盘一致性原则

回测、模拟盘和实盘应尽量复用相同的策略逻辑、仓位逻辑、风控规则和成本模型，减少“回测可行但实盘不可执行”的问题。

#### 1.4.4 风控前置原则

所有真实订单在提交券商接口前必须经过风控检查。风控检查失败时，订单不得提交。

#### 1.4.5 状态可追溯原则

策略信号、交易计划、风控结果、订单请求、券商返回、成交回报、撤单请求、人工操作都必须记录日志，保证每一笔实盘交易可以追溯。

#### 1.4.6 异常停止原则

当系统出现行情异常、券商连接异常、账户同步异常、订单状态未知、风控异常或人工停止指令时，系统应立即停止自动下单。

#### 1.4.7 可扩展但不高频化原则

系统允许扩展新的低频策略、指标、因子、券商接口和风控规则，但不允许扩展为高频交易系统。

## 2. 系统总体架构

1. 接口层
2. 应用服务层
3. 策略与研究层
4. 交易执行层
5. 风控与规则层
6. 数据访问层

### 2.2 逻辑分层说明

#### 2.2.1 API 接口层

API 接口层负责对外提供 RESTful API 服务，包括：

* 数据接口
* 策略接口
* 回测接口
* 参数优化接口
* 模拟盘接口
* 实盘账户接口
* 实盘订单接口
* 风控接口
* 报告接口
* 监控接口

API 层只负责参数校验、权限预留、请求转发和响应封装，不应包含复杂业务逻辑。

#### 2.2.2 应用服务层

应用服务层负责组织业务流程，例如：

* 创建回测任务
* 创建参数优化任务
* 启动模拟盘策略实例
* 启动或停止实盘策略实例
* 生成交易计划
* 同步实盘账户
* 查询报告结果

应用服务层负责协调多个底层模块，但不直接处理具体指标计算、订单提交或数据库细节。

#### 2.2.3 策略与研究层

策略与研究层负责量化研究相关逻辑，包括：

* 指标计算
* 因子计算
* 策略信号生成
* 策略参数管理
* 策略版本管理
* 参数优化
* 训练数据集预留

该层不允许直接调用券商接口。

#### 2.2.4 回测与模拟盘层

回测与模拟盘层负责策略验证，包括：

* 历史回测
* 模拟账户
* 模拟订单
* 模拟成交
* 模拟持仓
* 回测报告
* 模拟盘报告

回测层和模拟盘层应尽量复用策略信号、仓位管理和风控逻辑。

#### 2.2.5 交易执行层

交易执行层负责实盘交易链路，包括：

* 交易计划
* 订单创建
* 订单提交
* 订单状态同步
* 成交回报处理
* 持仓更新
* 资金同步
* 撤单处理

交易执行层必须依赖风控层进行订单前置检查。

#### 2.2.6 风控与规则层

风控与规则层负责：

* A 股交易规则校验
* 仓位上限检查
* 资金检查
* 可卖数量检查
* 涨跌停检查
* 停牌检查
* T+1 检查
* 交易频率限制
* 单日亏损限制
* 最大回撤限制
* 一键停止交易

风控与规则层是实盘交易安全的核心。

#### 2.2.7 数据访问层

数据访问层负责数据库读写，包括：

* 股票行情数据
* 指标数据
* 因子数据
* 策略参数
* 回测任务与结果
* 模拟盘账户
* 实盘订单
* 实盘成交
* 持仓快照
* 资金快照
* 风控事件
* 审计日志

该层应通过 Repository 或 DAO 模式封装数据库访问。

#### 2.2.8 基础设施层

基础设施层包括：

* PostgreSQL
* Redis
* Celery 或 Dramatiq
* 日志系统
* 配置管理
* 定时任务
* Docker 部署环境

#### 2.2.9 监控与审计层

监控与审计层横跨系统各模块，负责：

* 系统健康状态监控
* 行情连接监控
* 券商连接监控
* 账户状态监控
* 订单异常监控
* 策略频率监控
* 风控事件记录
* 审计日志记录
* 告警通知

## 3. 核心模块设计

## 3.1 数据管理模块设计

### 3.1.1 模块职责

数据管理模块负责系统基础数据的导入、清洗、存储和查询，是策略研究和回测的基础。

主要职责包括：

1. 股票基础信息管理。
2. 日线行情数据管理。
3. 可选分钟级行情数据管理。
4. 复权因子管理。
5. 停牌状态管理。
6. 涨跌停状态管理。
7. A 股交易日历管理。
8. 数据质量检查。

### 3.1.2 子模块划分

```text
data/
├── stock_info_service       股票基础信息服务
├── daily_data_service       日线行情服务
├── minute_data_service      可选分钟行情服务
├── calendar_service         交易日历服务
├── data_import_service      数据导入服务
├── data_cleaner             数据清洗器
└── data_repository          数据访问对象
```

### 3.1.3 数据导入流程

```text
上传/指定数据文件
        ↓
字段格式校验
        ↓
股票代码和日期规范化
        ↓
重复数据检查
        ↓
缺失数据检查
        ↓
异常价格检查
        ↓
停牌/涨跌停标记
        ↓
写入数据库
        ↓
记录导入日志
```

### 3.1.4 数据质量控制

系统应至少检查以下问题：

1. 股票代码为空。
2. 交易日期为空。
3. 开高低收价格为空。
4. 成交量为负。
5. 最高价低于最低价。
6. 收盘价超出最高价和最低价区间。
7. 同一股票同一日期重复记录。
8. 非交易日出现交易数据。
9. 复权因子缺失或异常。

### 3.1.5 高频数据排除设计

系统不设计 Tick 数据表，不建设 Tick 数据导入接口，不将 Tick 数据作为策略输入。可选分钟级行情仅用于低频策略辅助执行、实盘价格确认和简单盘中状态判断。

## 3.2 指标计算模块设计

### 3.2.1 模块职责

指标计算模块负责根据行情数据计算技术指标，为策略信号和因子计算提供输入。

主要指标包括：

* MA
* EMA
* MACD
* RSI
* BOLL
* ATR
* 成交量均线
* 收益率
* 波动率

### 3.2.2 子模块划分

```text
indicator/
├── trend_indicator          趋势类指标
├── momentum_indicator       动量类指标
├── volatility_indicator     波动率指标
├── volume_indicator         成交量指标
├── indicator_pipeline       指标计算流水线
└── indicator_repository     指标结果存储
```

### 3.2.3 指标计算流程

```text
读取行情数据
        ↓
按股票代码分组
        ↓
按日期排序
        ↓
计算指标
        ↓
处理前置空值
        ↓
保存指标结果
        ↓
返回给策略模块
```

### 3.2.4 设计要求

1. 指标计算必须按时间顺序执行。
2. 指标不得使用未来数据。
3. 指标参数必须可配置。
4. 指标结果应可缓存或持久化。
5. 指标计算应支持批量处理。

## 3.3 因子计算模块设计

### 3.3.1 模块职责

因子计算模块负责生成用于选股、打分和机器学习预留的低频量化特征。

主要因子包括：

* 5 日收益率
* 20 日收益率
* 60 日收益率
* 短期反转因子
* 成交量变化因子
* 波动率因子
* 趋势强度因子
* 最大回撤因子

### 3.3.2 子模块划分

```text
factor/
├── momentum_factor          动量因子
├── reversal_factor          反转因子
├── volatility_factor        波动率因子
├── volume_factor            成交量因子
├── risk_factor              风险因子
├── factor_score             综合因子评分
└── factor_repository        因子结果存储
```

### 3.3.3 因子计算流程

```text
读取行情和指标数据
        ↓
计算单因子
        ↓
因子缺失值处理
        ↓
因子标准化
        ↓
因子方向调整
        ↓
综合评分计算
        ↓
保存因子结果
```

### 3.3.4 因子标准化设计

因子标准化可采用以下方式：

1. Z-Score 标准化。
2. 分位数标准化。
3. 去极值处理。
4. 行业中性化预留。
5. 市值中性化预留。

第一版可先实现 Z-Score 标准化和去极值处理。

## 3.4 策略管理模块设计

### 3.4.1 模块职责

策略管理模块负责低频策略的定义、配置、版本管理、启停控制和信号生成。

策略模块只输出以下结果：

* 买入信号
* 卖出信号
* 减仓信号
* 清仓信号
* 目标仓位
* 目标持股列表

策略模块不得直接下单。

### 3.4.2 子模块划分

```text
strategy/
├── base_strategy            策略基类
├── trend_strategy           趋势跟踪策略
├── mean_reversion_strategy  均值回归策略
├── factor_score_strategy    因子打分策略
├── signal_generator         信号生成器
├── strategy_param_service   策略参数服务
├── strategy_version_service 策略版本服务
└── strategy_repository      策略数据访问
```

### 3.4.3 策略基类设计

策略基类应定义统一接口：

```text
initialize(params)
prepare_data(context)
generate_signals(context)
generate_target_position(context)
validate_params(params)
```

所有策略都必须继承策略基类，确保回测、模拟盘和实盘能够统一调用。

### 3.4.4 策略信号结构

策略信号建议包含以下字段：

```text
signal_id
strategy_id
strategy_instance_id
symbol
signal_date
signal_type
signal_strength
target_position
reason
params_version
generated_at
```

其中 signal_type 可包括：

```text
buy
sell
reduce
clear
hold
```

### 3.4.5 策略频率限制

系统应从架构上限制策略执行频率，避免低频策略因错误配置或程序 bug 变成高频策略。

频率限制包括：

1. 策略最小执行间隔。
2. 单策略每日最大信号数。
3. 单策略每日最大订单数。
4. 单股票每日最大交易次数。
5. 短时间重复信号过滤。

### 3.4.6 策略版本管理

策略每次参数变更都应产生参数版本。回测、模拟盘和实盘记录必须关联策略版本和参数版本，保证交易结果可追溯。

## 3.5 仓位管理模块设计

### 3.5.1 模块职责

仓位管理模块负责根据策略目标、账户资金、风险限制和 A 股交易规则生成可执行的目标交易数量。

主要职责包括：

1. 计算目标仓位。
2. 计算买入金额。
3. 计算卖出数量。
4. 控制单票最大仓位。
5. 控制总仓位上限。
6. 控制最低现金比例。
7. 控制最大持股数量。
8. 处理 A 股 100 股整数倍规则。

### 3.5.2 仓位计算流程

```text
读取策略目标仓位
        ↓
读取账户资金和持仓
        ↓
应用仓位限制
        ↓
应用现金保留限制
        ↓
应用单票上限限制
        ↓
应用最大持股数量限制
        ↓
生成目标买卖金额
        ↓
转换为订单数量
        ↓
应用 A 股交易单位规则
```

### 3.5.3 仓位参数

```text
initial_position_ratio       首次买入比例
max_position_per_stock       单票最大仓位
max_total_position           总仓位上限
min_cash_ratio               最低现金比例
max_hold_stock_count         最大持股数量
position_add_ratio           加仓比例
position_reduce_ratio        减仓比例
```

## 3.6 回测引擎模块设计

### 3.6.1 模块职责

回测引擎模块负责使用历史行情模拟策略执行过程，输出交易记录、资产曲线和绩效指标。

回测引擎以日线回测为核心，可扩展低频分钟级回测，但不支持 Tick 级回测。

### 3.6.2 子模块划分

```text
backtest/
├── backtest_engine          回测主引擎
├── backtest_context         回测上下文
├── backtest_broker          模拟撮合器
├── backtest_account         回测账户
├── backtest_position        回测持仓
├── backtest_order           回测订单
├── backtest_metrics         回测指标
└── backtest_report          回测报告
```

### 3.6.3 回测时间推进机制

日线回测采用按交易日推进：

```text
初始化账户
        ↓
按交易日循环
        ↓
读取当日行情
        ↓
更新持仓市值
        ↓
生成策略信号
        ↓
执行回测风控
        ↓
生成模拟订单
        ↓
模拟成交
        ↓
更新现金和持仓
        ↓
记录每日资产
        ↓
进入下一交易日
```

### 3.6.4 成交规则设计

第一版可支持以下成交规则：

1. 次日开盘价成交。
2. 当日收盘价成交。
3. 指定价格成交。
4. 加入滑点后的价格成交。

推荐默认使用“次日开盘价成交”，以降低未来函数风险。

### 3.6.5 A 股规则模拟

回测中必须模拟：

1. 非交易日不交易。
2. 停牌日不交易。
3. 涨停时买入可能失败。
4. 跌停时卖出可能失败。
5. 当日买入股票 T+1 后才可卖出。
6. 买入数量为 100 股整数倍。
7. 卖出数量不得超过可卖数量。
8. 交易成本包括佣金、印花税和滑点。

### 3.6.6 回测指标

回测结果应至少包含：

* 总收益率
* 年化收益率
* 最大回撤
* 夏普比率
* 卡玛比率
* 胜率
* 盈亏比
* 交易次数
* 平均持仓天数
* 换手率
* 总手续费
* 最终资产

## 3.7 参数优化模块设计

### 3.7.1 模块职责

参数优化模块负责在参数搜索空间内寻找综合评分更优的策略参数。

### 3.7.2 子模块划分

```text
optimizer/
├── param_space              参数空间定义
├── grid_search              网格搜索
├── random_search            随机搜索
├── optuna_optimizer         贝叶斯优化
├── objective_function       目标函数
├── trial_runner             单轮试验执行器
└── optimize_repository      优化结果存储
```

### 3.7.3 优化流程

```text
创建优化任务
        ↓
读取参数搜索空间
        ↓
生成参数组合
        ↓
执行回测
        ↓
计算绩效指标
        ↓
计算综合评分
        ↓
保存 trial 结果
        ↓
判断是否继续
        ↓
输出最优参数
        ↓
执行样本外验证
```

### 3.7.4 目标函数设计

默认目标函数：

```text
score = annual_return * 0.35
      + sharpe_ratio * 0.25
      - max_drawdown * 0.25
      - turnover_penalty * 0.10
      - concentration_penalty * 0.05
```

其中换手率惩罚用于避免优化器找到过度交易或异常高频交易倾向的参数组合。

### 3.7.5 防过拟合设计

参数优化必须支持：

1. 样本内训练。
2. 样本外验证。
3. 多时间段验证。
4. 多股票池验证。
5. 换手率惩罚。
6. 最大回撤约束。
7. 单票集中度惩罚。
8. 交易频率约束。

## 3.8 模拟盘模块设计

### 3.8.1 模块职责

模拟盘模块用于在不使用真实资金的情况下验证策略在接近实盘环境中的运行效果。

模拟盘是实盘前的必要阶段。

### 3.8.2 子模块划分

```text
simulation/
├── simulation_account       模拟账户
├── simulation_order         模拟订单
├── simulation_trade         模拟成交
├── simulation_position      模拟持仓
├── simulation_engine        模拟盘引擎
└── simulation_report        模拟盘报告
```

### 3.8.3 模拟盘运行流程

```text
启动模拟盘策略实例
        ↓
读取交易日历
        ↓
按低频调度执行策略
        ↓
读取实时或准实时行情
        ↓
生成交易计划
        ↓
执行模拟盘风控
        ↓
生成模拟订单
        ↓
按照模拟成交规则成交
        ↓
更新模拟账户资金和持仓
        ↓
记录模拟盘日志
```

### 3.8.4 模拟成交规则

模拟成交可采用以下规则：

1. 使用最新价成交。
2. 使用买一或卖一价格成交。
3. 使用指定滑点成交。
4. 涨停买入失败。
5. 跌停卖出失败。
6. 停牌不成交。

### 3.8.5 模拟盘与回测区别

回测基于历史数据一次性运行；模拟盘基于真实交易日逐日运行。模拟盘更接近实盘执行流程，可以测试调度、行情、风控、订单状态和日志链路。

## 3.9 实盘账户模块设计

### 3.9.1 模块职责

实盘账户模块负责从券商接口同步真实账户资金和持仓信息。

主要职责包括：

1. 查询总资产。
2. 查询可用资金。
3. 查询冻结资金。
4. 查询持仓。
5. 查询可卖数量。
6. 保存资金快照。
7. 保存持仓快照。
8. 下单前强制刷新账户状态。

### 3.9.2 账户同步流程

```text
调用券商账户查询接口
        ↓
获取资金信息
        ↓
获取持仓信息
        ↓
字段标准化
        ↓
写入账户快照
        ↓
写入持仓快照
        ↓
返回最新账户状态
```

### 3.9.3 账户状态异常处理

如果账户同步失败，系统应：

1. 标记账户状态异常。
2. 记录异常日志。
3. 发送告警。
4. 禁止自动下单。
5. 等待人工确认或自动重试。

## 3.10 券商接口适配模块设计

### 3.10.1 模块职责

券商接口适配模块负责屏蔽不同券商或交易网关的接口差异，向系统内部提供统一交易接口。

### 3.10.2 适配器模式设计

系统采用 BrokerAdapter 抽象接口：

```text
BrokerAdapter
├── connect()
├── disconnect()
├── get_connection_status()
├── query_account()
├── query_positions()
├── query_orders()
├── query_trades()
├── place_order()
└── cancel_order()
```

不同券商接口通过实现 BrokerAdapter 接入系统。

### 3.10.3 统一订单结构

内部订单结构应与券商返回结构解耦，避免系统被某个券商接口绑定。

统一订单字段包括：

```text
symbol
side
price
quantity
order_type
strategy_instance_id
plan_id
signal_id
client_order_id
```

### 3.10.4 接口异常处理

券商接口异常包括：

1. 登录失败。
2. 连接断开。
3. 查询超时。
4. 下单超时。
5. 撤单失败。
6. 返回字段异常。
7. 订单状态未知。

出现异常时，应停止自动下单并记录审计日志。

## 3.11 实盘订单模块设计

### 3.11.1 模块职责

实盘订单模块负责订单创建、风控状态记录、提交、同步、撤单和状态恢复。

### 3.11.2 订单状态机

```text
created
   ↓
risk_checked
   ↓
submitted
   ↓
accepted
   ↓───────────────┐
partially_filled   │
   ↓               │
filled             │
                   │
cancel_requested ──┤
   ↓               │
canceled           │
                   │
rejected           │
failed             │
unknown            │
```

### 3.11.3 订单创建流程

```text
读取交易计划
        ↓
生成候选订单
        ↓
执行订单风控
        ↓
风控通过
        ↓
创建本地订单
        ↓
提交券商接口
        ↓
记录券商返回
        ↓
更新订单状态
```

### 3.11.4 防重复下单机制

系统应通过以下方式防止重复下单：

1. 交易计划唯一编号。
2. 策略信号唯一编号。
3. 客户端订单编号。
4. 同一策略、同一股票、同一方向、同一交易计划不得重复提交。
5. 订单提交前检查未完成订单。
6. 程序重启后先同步订单状态再允许继续下单。

### 3.11.5 订单状态未知处理

当订单状态未知时，系统应：

1. 停止该策略自动下单。
2. 查询券商当日委托。
3. 查询券商当日成交。
4. 尝试匹配本地订单。
5. 如果无法确认，标记为 unknown。
6. 发送告警。
7. 等待人工处理。

## 3.13 实盘风控模块设计

### 3.13.1 模块职责

实盘风控模块负责在真实订单提交前进行检查。风控模块必须独立于策略模块存在。

### 3.13.2 风控分层

实盘风控分为三层：

```text
账户级风控
策略级风控
订单级风控
```

#### 账户级风控

包括：

1. 账户总仓位上限。
2. 最低现金保留比例。
3. 单日最大亏损限制。
4. 账户最大回撤限制。
5. 一键停止交易。

#### 策略级风控

包括：

1. 单策略资金上限。
2. 单策略持股数量上限。
3. 单策略当日亏损限制。
4. 策略连续亏损暂停。
5. 策略交易频率限制。

#### 订单级风控

包括：

1. 买入金额不得超过可用资金。
2. 卖出数量不得超过可卖持仓。
3. 订单价格不得超出涨跌停范围。
4. 股票不得停牌。
5. 数量必须符合交易单位规则。
6. 不得重复下单。
7. 单笔订单金额不得超过上限。
8. 当日订单数量不得超过上限。

### 3.13.3 风控检查流程

```text
候选订单
   ↓
交易日检查
   ↓
交易时段检查
   ↓
行情状态检查
   ↓
账户状态检查
   ↓
持仓和资金检查
   ↓
A 股规则检查
   ↓
仓位限制检查
   ↓
频率限制检查
   ↓
风险阈值检查
   ↓
通过 / 拒绝
```

### 3.13.4 风控结果结构

风控结果应包含：

```text
risk_check_id
order_id
passed
risk_items
reject_reason
action
checked_at
```

### 3.13.5 风控失败处理

风控失败时：

1. 不提交券商接口。
2. 记录风控事件。
3. 记录审计日志。
4. 必要时发送告警。
5. 若为严重风险，暂停策略或停止全部交易。

## 6. 任务调度设计

### 6.1 任务类型

系统中存在以下任务类型：

1. 数据导入任务。
2. 指标计算任务。
3. 因子计算任务。
4. 回测任务。
5. 参数优化任务。
6. 模拟盘调度任务。
7. 实盘账户同步任务。
8. 实盘订单同步任务。
9. 监控检查任务。
10. 报告生成任务。

### 6.2 同步任务与异步任务划分

#### 同步任务

适合同步执行的任务：

* 查询股票列表
* 查询回测结果
* 查询账户快照
* 查询订单状态
* 查询报告
* 单次风控检查

#### 异步任务

适合异步执行的任务：

* 数据导入
* 批量指标计算
* 大规模因子计算
* 回测
* 参数优化
* 报告生成
* 批量账户同步

### 6.3 定时任务设计

定时任务包括：

1. 每日收盘后更新行情。
2. 每日收盘后计算指标和因子。
3. 每日收盘后生成次日交易计划。
4. 交易日前同步账户。
5. 交易时段内定时同步订单和成交。
6. 交易时段内监控行情和交易接口状态。
7. 交易日结束后生成实盘日报。

### 6.4 任务状态

任务状态统一设计为：

```text
pending
running
success
failed
canceled
```

失败任务必须记录错误原因和堆栈摘要。

## 7. 数据库概要设计

### 7.1 数据库选型

系统主数据库建议采用 PostgreSQL，原因如下：

1. 支持复杂查询。
2. 支持事务。
3. 支持 JSONB 字段。
4. 适合结构化金融数据存储。
5. 便于存储策略参数和风控配置。

Redis 用于缓存和任务状态辅助，不作为核心交易数据的唯一存储。

### 7.2 核心数据域

系统数据可分为以下数据域：

1. 行情数据域。
2. 指标因子数据域。
3. 策略数据域。
4. 回测数据域。
5. 优化数据域。
6. 模拟盘数据域。
7. 实盘账户数据域。
8. 实盘订单成交数据域。
9. 风控事件数据域。
10. 审计日志数据域。

### 7.3 核心表关系概述

```text
strategy
   └── strategy_params
          └── strategy_instance
                 ├── backtest_task
                 ├── simulation_order
                 └── live_order

live_order
   └── live_trade

broker_account
   ├── live_account_snapshot
   ├── live_position_snapshot
   ├── live_order
   └── live_trade

backtest_task
   ├── backtest_result
   ├── backtest_trade_record
   └── portfolio_daily
```

### 7.4 索引设计原则

应建立索引的字段包括：

1. symbol。
2. trade_date。
3. strategy_id。
4. strategy_instance_id。
5. task_id。
6. order_status。
7. broker_order_id。
8. created_at。
9. trade_time。

行情表应建立 `(symbol, trade_date)` 唯一索引。

实盘订单表应建立 `client_order_id` 或等效字段唯一索引，防止重复下单。

### 7.5 事务一致性设计

涉及实盘订单、成交、资金和持仓更新时，应使用事务保证一致性。

尤其是：

1. 订单状态更新与成交记录写入。
2. 成交记录写入与持仓更新。
3. 风控结果记录与订单创建。
4. 一键停止状态更新与审计日志写入。

## 9. 安全设计

### 9.1 账户安全

券商账户、密码、密钥等敏感信息不得明文存储在代码或日志中。配置应通过环境变量、加密配置或安全配置服务管理。

### 9.2 实盘开关

系统应设置全局实盘交易开关，默认关闭。

启用实盘交易至少应满足：

1. 券商接口连接正常。
2. 风控配置已启用。
3. 策略实例状态正常。
4. 账户同步成功。
5. 操作者显式开启实盘模式。

### 9.3 权限预留

第一版可不实现复杂权限，但接口设计应预留：

1. 管理员权限。
2. 策略操作权限。
3. 实盘交易权限。
4. 风控配置权限。
5. 审计查询权限。

### 9.4 操作审计

以下操作必须审计：

1. 启动实盘。
2. 停止实盘。
3. 修改风控参数。
4. 修改策略参数。
5. 下单。
6. 撤单。
7. 一键停止。

## 10. 异常处理设计

### 10.1 异常分类

系统异常分为：

1. 数据异常。
2. 策略异常。
3. 回测异常。
4. 优化异常。
5. 行情异常。
6. 券商接口异常。
7. 账户同步异常。
8. 订单状态异常。
9. 风控异常。
10. 系统运行异常。

### 10.2 异常处理原则

1. 研究和回测异常不影响实盘模块。
2. 实盘相关异常应优先保证不继续下单。
3. 订单状态未知时不得继续自动交易。
4. 风控模块异常时不得下单。
5. 行情不可用时不得下单。
6. 账户资金或持仓不同步时不得下单。

### 10.3 关键异常处理表

| 异常类型   | 处理方式          |
| ------ | ------------- |
| 行情连接断开 | 停止自动下单，发送告警   |
| 券商接口断开 | 停止自动下单，尝试重连   |
| 账户同步失败 | 停止自动下单，记录异常   |
| 订单状态未知 | 暂停策略，执行订单恢复流程 |
| 风控检查异常 | 拒绝订单，发送告警     |
| 策略执行异常 | 暂停策略，记录错误     |
| 重复下单风险 | 拒绝订单，记录风控事件   |
| 交易频率异常 | 暂停策略，发送告警     |

## 11. 低频约束设计

### 11.1 低频定位约束

系统应从架构、数据、策略、任务调度和风控多个层面保证项目不演化为高频系统。

### 11.2 数据层约束

1. 不设计 Tick 数据表。
2. 不建设 Tick 数据导入链路。
3. 不使用逐笔成交数据生成策略信号。
4. 分钟级行情仅作为可选辅助数据。

### 11.3 策略层约束

1. 策略必须声明 frequency。
2. 策略执行频率必须受限。
3. 不允许策略在极短时间内循环触发下单。
4. 不允许策略直接调用订单接口。

### 11.4 订单层约束

1. 限制单策略每日订单数量。
2. 限制单股票每日交易次数。
3. 限制短时间重复订单。
4. 不支持高频批量报单。
5. 不支持高频撤单策略。

### 11.5 风控层约束

1. 监控策略信号频率。
2. 监控订单提交频率。
3. 超过频率限制时暂停策略。
4. 将频率异常记录为风控事件。

## 13. 技术选型设计

### 13.1 后端框架

## 16. 总结

本文档从总体架构、模块划分、核心流程、数据流、任务调度、数据库概要、API 概要、安全、异常处理、低频约束和部署等方面，对 A 股低频量化策略研究、回测、模拟盘与实盘执行系统进行了概要设计。

系统的核心设计思想是：

1. 以低频量化交易为长期定位。
2. 以回测、模拟盘和实盘一致性为基础。
3. 以实盘风控和审计追踪为安全边界。
4. 以模块化架构支持后续扩展。
5. 明确排除高频交易、Tick 策略和 Level-2 盘口策略。

该概要设计可作为后续数据库设计、接口设计、策略设计、风控设计和编码实现的基础。
